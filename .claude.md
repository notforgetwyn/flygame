# 战争飞机雷霆 - 项目蓝图

## 1. 协作规则

本项目按以下规则推进：

1. 先输出简洁但专业的需求分析文档。
2. 再输出项目架构设计，包括项目目录结构、项目架构设计、页面设计、UI 设计、模块划分、类设计、数据流设计。
3. 然后把项目拆成多个迭代版本：MVP、道具系统、敌机AI与关卡、存档与设置、优化。
4. 每次只完成一个小阶段，不一次性写完整个项目。
5. 每个阶段都先说明目标、实现思路、涉及文件，再给代码。
6. 所有代码必须可运行、结构清晰、便于扩展，禁止把所有逻辑写进一个文件。
7. 优先使用面向对象设计，加入必要注释，但不过度冗余。
8. 配置和存档统一使用 JSON 文件。
9. 写代码时考虑异常处理、边界情况和后续扩展。
10. 每完成一个阶段，都给出下一阶段建议。

额外约束：

- 先从最小可运行版本开始：玩家战机移动、发射子弹、敌机生成与下落、碰撞检测、得分、游戏结束。
- 然后再逐步加入道具系统、敌机AI、关卡设计。
- 用户说"继续下一步"后，才进入下一个阶段。
- 用户说"重构"后，按工程化方式优化当前结构。
- 用户说"解释"后，用适合初学者的方式解释代码。

## 2. 需求分析

### 2.1 项目目标

开发一个基于 Python 的竖版飞行射击游戏（类似雷电），技术栈不限定，可使用终端/TUI、控制台绘图或轻量 GUI 框架，具备可玩性、可维护性和可扩展性，并按照工程化迭代方式逐步建设。

### 2.2 核心功能需求

1. 玩家战机支持上下左右移动。
2. 玩家战机可发射子弹。
3. 敌机从上方生成并向下滑落。
4. 子弹击中敌机后敌机爆炸并计分。
5. 敌机撞击玩家战机后玩家损失生命值。
6. 游戏过程中实时显示分数、关卡和生命值。
7. 游戏启动后先进入开始界面，而不是直接开始。
8. 开始界面支持：
   - 开始游戏
   - 继续游戏
   - 设置
   - 退出游戏
9. 支持游戏存档与读档。
10. 设置页面支持调整难度、音效音量，并持久化保存。
11. 整体架构支持后续扩展，例如道具系统、敌机AI、关卡设计、排行榜、Boss战等功能。

### 2.3 非功能需求

1. 代码结构清晰，模块职责明确。
2. 采用面向对象设计。
3. 配置与存档使用 JSON 存储。
4. 具备基本异常处理能力，例如配置文件缺失、存档损坏、非法数据回退默认值。
5. 便于测试和后续重构。

### 2.4 边界与风险

1. 玩家战机移动需要限制边界，避免超出游戏区域。
2. 敌机生成位置需要随机但避免重叠。
3. 子弹数量需要限制，避免无限发射导致性能问题。
4. 配置文件或存档文件不存在时，需要自动创建或回退默认值。
5. 读档时需要校验数据合法性，避免程序崩溃。
6. 碰撞检测需要精确，避免穿模或误判。
7. 随等级提升，敌机生成速度和移动速度需要设置合理上限。

#### 分层说明

1. UI 展示层
   - 负责页面展示、按钮交互、文本显示和 HUD 渲染。
   - 不直接持久化数据，不承担底层文件读写职责。

2. 应用控制层
   - 负责程序启动、图形/终端初始化、主循环管理、事件分发和场景切换。
   - 连接 UI 展示层与业务逻辑层，是运行时调度中心。

3. 业务逻辑层
   - 负责玩家移动、子弹发射、敌机生成、碰撞检测、爆炸效果、计分规则等核心规则。
   - 尽量不依赖具体页面，以保证后续可测试和可复用。

4. 配置与存档服务层
   - 负责设置和存档的读取、保存、校验、默认值补全和异常兜底。
   - 向上为场景和应用层提供统一的数据访问接口。

5. 文件管理层
   - 负责组织配置目录、存档目录、资源目录和文档目录。
   - 屏蔽具体文件路径，避免业务代码散落硬编码路径。

6. 数据存储层
   - 当前阶段采用 JSON，适合设置、存档和轻量数据持久化。
   - 后续如果出现排行榜、成就或复杂用户数据，再考虑升级到 SQLite。

7. 日志与诊断预留层
   - 当前版本先预留接口和位置。
   - 后续可加入日志记录、错误诊断和调试开关，提升可维护性。

#### 架构设计原则

1. UI 与业务逻辑解耦，避免把游戏规则直接写死在页面渲染代码中。
2. 配置与存档统一通过服务层访问，避免后续功能增加时重复读写 JSON。
3. 主循环只负责调度，不承载过多业务细节。
4. 各层职责单一，便于后续加入道具系统、敌机AI、排行榜和主题皮肤。

### 3.3 页面与 UI 对应设计

#### 启动流程

1. 启动程序
2. 进入主菜单页
3. 从主菜单进入开始游戏 / 继续游戏 / 设置 / 退出

#### Page 1: 主菜单页 `MenuScene`

页面目标：
- 作为程序统一入口
- 引导玩家开始新游戏或进入其他功能页

UI 组成：
- 顶部标题：`战争飞机雷霆`
- 中部菜单项：
  - 开始游戏
  - 继续游戏
  - 设置
  - 退出游戏
- 底部操作提示：方向键选择，`Enter` 确认

布局设计：
- 标题位于页面上方居中
- 菜单按钮在垂直方向居中排列
- 当前选中项使用高亮色
- 底部显示轻量提示文本

交互设计：
- `Up/Down` 或 `W/S` 切换选项
- `Enter/Space` 执行当前选项
- `ESC` 退出程序

视觉设计：
- 背景：深色配星空效果
- 标题：金色大字号
- 普通菜单项：白色
- 选中菜单项：橙色高亮

#### Page 2: 游戏页 `GameplayScene`

页面目标：
- 承载核心玩法
- 展示实时游戏状态

UI 组成：
- 游戏区域（宽 x 高）
- 顶部 HUD：
  - 分数
  - 关卡
  - 生命值/血条
- 玩家战机（底部区域）
- 敌机（从上方生成）
- 子弹（玩家发射向上）
- 爆炸效果

布局设计：
- 游戏区域居中
- HUD 固定顶部显示
- 玩家战机位于底部 1/4 区域
- 敌机从顶部向下移动

交互设计：
- `Left/Right` 或 `A/D` 控制左右移动
- `Up/Down` 或 `W/S` 控制上下移动
- `Space` 或 `Z` 发射子弹
- `ESC` 暂停并返回主菜单
- 游戏结束后按 `R` 重新开始

视觉设计：
- 背景：星空滚动效果
- 玩家战机：银色/蓝色战斗机
- 敌机：红色/灰色不同类型
- 子弹：黄色/蓝色光弹
- 爆炸：橙色/红色粒子效果
- HUD 文本：白色带描边

#### Page 3: 设置页 `SettingsScene`

页面目标：
- 调整游戏难度和音量
- 保存用户配置

UI 组成：
- 页面标题：`设置`
- 难度设置项：
  - 敌机速度
  - 敌机生成频率
- 音效音量调节
- 操作按钮：
  - 保存设置
  - 返回主菜单

布局设计：
- 标题顶部居中
- 中部展示各项设置
- 底部展示保存和返回按钮

交互设计：
- 左右键或按钮调整数值
- `Enter` 保存当前配置
- `ESC` 返回菜单

视觉设计：
- 与主菜单保持统一风格
- 当前选中控件高亮
- 数值用滑块或数字显示

#### Page 4: 继续游戏页 `ContinueScene`

页面目标：
- 从存档恢复游戏
- 在无存档时提供清晰反馈

UI 组成：
- 页面标题：`继续游戏`
- 存档状态文本：
  - 有存档：显示存档信息（分数、关卡）
  - 无存档：显示暂无存档
- 操作按钮：
  - 继续
  - 返回主菜单

布局设计：
- 标题居中
- 中部显示存档说明
- 下方放置可执行操作

交互设计：
- `Enter` 尝试继续
- `ESC` 返回菜单

视觉设计：
- 成功状态用白色/绿色
- 无存档用黄色提醒

#### Page 5: 结束页 `GameOverScene`

页面目标：
- 展示本局结果
- 提供后续操作入口

UI 组成：
- 标题：`游戏结束`
- 最终得分
- 到达关卡
- 操作按钮：
  - 重新开始
  - 返回主菜单

布局设计：
- 标题和得分居中显示
- 操作项位于得分下方

交互设计：
- `R` 重新开始
- `ESC` 返回菜单

视觉设计：
- 结束标题红色突出
- 得分与操作说明使用白色文本

### 3.4 全局 UI 规范

为保证每个页面风格统一，定义以下通用规范：

1. 颜色规范
   - 背景：深蓝/黑色配星空
   - 主文本：白色
   - 强调色：橙色/金色
   - 警告/结束：红色
   - 玩家战机：银色/蓝色
   - 敌机：红色/灰色/绿色（不同类型）
   - 子弹：黄色/蓝色
   - 爆炸：橙色/红色渐变

2. 布局规范
   - 页面主内容尽量居中
   - HUD 固定顶部显示，不遮挡核心玩法
   - 菜单类页面采用垂直排列

3. 组件规范
   - 文本组件：标题、正文、提示三种字号
   - 按钮组件：默认态、选中态、禁用态
   - 血条组件：绿色→黄色→红色渐变

4. 扩展预留
   - 后续可加入动效、音效、粒子系统和主题皮肤

#### 模块职责说明

1. 程序入口模块
   - 只负责启动，不承载业务逻辑。

2. 应用控制模块
   - 负责把页面、事件、时钟和主循环组织起来。

3. 页面场景模块
   - 每个页面单独成模块，负责该页面的展示和交互。

4. 业务模型模块
   - 管理玩家、子弹、敌机、道具、爆炸效果、分数、关卡等核心数据和规则。

5. 配置与存档服务模块
   - 提供统一 JSON 读写、默认值回退、容错处理。

6. UI 组件模块
   - 抽离可复用控件，避免每个场景重复写按钮和文本逻辑。

7. 常量与配置模块
   - 集中管理全局配置，防止魔法数字散落。

#### 类职责说明

5. `SettingsScene`
   - 负责难度设置展示、修改和保存。

   - 负责读取存档状态并决定是否恢复游戏。

7. `GameOverScene`

### 3.7 数据流设计

```text
用户输入
   ↓
当前场景接收事件
   ↓
应用控制层分发动作
   ↓
业务逻辑层更新游戏状态
   ↓
配置/存档服务层按需读写 JSON
   ↓
场景重新渲染 UI
```

#### 运行时数据流

```text
键盘输入
   ↓
GameplayScene.handle_event()
   ↓
Player.move() / Player.shoot()
   ↓
Bullet.update() / Enemy.update()
   ↓
CollisionSystem.check_collisions()
   ↓
Enemy.take_damage() / Player.take_damage()
   ↓
ScoreManager.add_score()
   ↓
GameplayScene.render()
   ↓
屏幕显示最新游戏状态
```

#### 配置数据流

```text
程序启动
   ↓
SettingsService.load_settings()
   ↓
读取 config/settings.json
   ↓
非法或缺失则回退默认值
   ↓
应用到游戏运行参数
   ↓
用户在设置页修改设置
   ↓
SettingsService.save_settings()
   ↓
写回 settings.json
```

#### 存档数据流

```text
游戏运行中
   ↓
GameSession / GameData 生成当前状态
   ↓
SaveService.save_game()
   ↓
写入 data/save.json
   ↓
主菜单点击继续游戏
   ↓
SaveService.load_save()
   ↓
恢复玩家位置、分数、关卡、敌机状态等
   ↓
GameplayScene 根据存档恢复运行
```

#### 数据流设计原则

1. 页面层只消费数据和触发动作，不直接操作底层文件。
2. 业务状态变化先发生在内存对象，再通过服务层决定是否持久化。
3. JSON 文件始终通过统一入口读写，避免多个模块各自维护格式。
4. 数据流方向尽量单向，降低后续调试和扩展成本。

##### 场景 A: 正常游戏流程

1. 玩家从菜单选择"开始游戏"
2. 系统清空旧存档（如有）
3. 系统初始化玩家战机
4. 系统开始生成敌机波次
5. 玩家控制战机移动和射击
6. 子弹击中敌机产生爆炸并增加分数
7. 敌机被消灭或穿过屏幕消失
8. 随时间/分数提升关卡，敌机变强
9. 敌机撞击玩家损失生命值
10. 生命值归零后游戏结束
11. 玩家查看得分，选择重新开始或返回菜单

##### 场景 B: 存档继续流程

1. 玩家从菜单选择"继续游戏"
2. 系统检查存档文件
3. 系统读取并校验存档数据

5. 玩家继续游戏

1. 玩家从菜单选择"设置"

3. 玩家调整难度等级和音量
4. 玩家选择保存
5. 系统写入配置文件
6. 返回主菜单

##### 场景 D: 异常处理流程

1. 存档文件不存在：提示无存档，返回菜单
2. 存档数据损坏：使用默认值，提示存档已损坏
3. 配置文件缺失：创建默认配置文件
4. 碰撞检测异常：触发安全模式，避免穿模
